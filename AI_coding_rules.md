# AI 코딩 작업 지침 (클로드 코드 터미널용)
> 던전 핸드 개발 경험에서 추출한 실전 규칙

---

## 1. 백업은 작업 전에 한다 (작업 후가 아니다)

수정을 시작하기 전에 반드시 현재 상태를 보존한다.
- 작업 시작 전: `git commit -m "before: [작업명]"`
- 대규모 수정 전: `git branch backup-[날짜]`로 별도 브랜치 생성
- 되돌리기: `git checkout -- [파일명]` (직전 커밋으로 복원) 또는 `git reset --hard HEAD` (전체 복원)

**이유**: 파일은 한 번 덮어쓰면 끝이다. git이 있으니 반드시 쓴다.

---

## 2. 한 세션에 수정은 5건 이내로 제한한다

컨텍스트가 길어질수록 AI의 판단 품질이 떨어진다.
- 수정 5건 -> lint/빌드 검증 -> 테스트 -> git commit -> 세션 종료
- "하는 김에 이것도"의 유혹을 경계한다
- 급하지 않은 건 WORK_LOG.md에 TODO로 기록하고 다음 세션으로 넘긴다

**이유**: 10건을 한번에 하다 실패하면 10건을 다시 해야 한다. 5건씩 두 번이 훨씬 안전하다.

---

## 3. 수정 1건마다 검증 1회

수정과 검증을 묶어서 한 단위로 취급한다.
- 코드 수정 -> lint/빌드 에러 확인 -> 관련 기능 동작 확인
- 검증 없이 다음 수정으로 넘어가지 않는다
- 밸런스 수치는 시뮬레이션으로 검증한다 (감으로 판단하지 않는다)
- JSX 파일은 수정 후 빌드 명령으로 문법 오류를 즉시 잡는다

**이유**: 문제를 발견하는 시점이 늦을수록 원인 추적이 어려워진다.

---

## 4. 설계 논의와 구현을 분리한다

구현 세션에서 설계를 바꾸지 않는다.
- 설계 변경이 필요하면 현재 작업을 커밋하고, 변경 사항을 WORK_LOG.md에 기록한 뒤 다음 세션에서 처리한다
- 구현 중 새로운 아이디어가 떠오르면 // TODO: [아이디어] 주석만 남기고 현재 작업을 먼저 완료한다
- 설계 논의는 채팅에서, 구현은 터미널에서 한다

**이유**: 논의하면서 동시에 코딩하면 범위가 끝없이 확장된다. 던전 핸드에서 밸런스 수정 4건으로 시작해 결국 17건이 된 것이 그 예시다.

---

## 5. 세션 시작/종료 루틴

**세션 시작 시:**
1. git pull (다른 컴퓨터에서 작업했을 경우)
2. CLAUDE.md 자동 읽기 확인
3. WORK_LOG.md에서 이전 세션 상태와 이번 세션 TODO 확인
4. git commit -m "before: [이번 작업명]"

**세션 종료 시:**
1. lint/빌드 에러 없는지 최종 확인
2. WORK_LOG.md에 이번 세션 완료 내역 기록
3. MISTAKES.md에 실수 있었으면 기록
4. git add . && git commit -m "[완료 내역 요약]" && git push

---

## 6. 문서 체계

| 파일 | 역할 | 갱신 시점 |
|------|------|-----------|
| CLAUDE.md | 프로젝트 구조, 핵심 규칙, 금지사항. 세션 시작 시 자동 로드됨 | 구조 변경 시 |
| WORK_LOG.md | 세션별 변경 이력 + 다음 세션 TODO | 매 세션 종료 시 |
| MISTAKES.md | 반복되는 실수 기록. [활성]/[해결됨] 태그 | 실수 발생 시 |

- CLAUDE.md는 프로젝트 루트에 위치해야 클로드 코드가 자동으로 읽는다
- WORK_LOG.md는 최근 5세션만 유지하고, 오래된 건 WORK_LOG_ARCHIVE.md로 이동한다
- MISTAKES.md는 항목 10개 초과 시 [해결됨] 항목을 정리한다

---

## 7. 테스트는 생략하지 않는다

코드가 이론상 맞아 보여도 실제로 돌려봐야 한다.
- 수정 후 반드시 해당 기능을 직접 테스트한다
- 엣지 케이스를 의식적으로 시도한다 (덱이 비었을 때, 카드가 1장일 때 등)
- lint 통과했으니 됐겠지는 충분하지 않다
- 가능하면 브라우저에서 직접 플레이해서 확인한다

---

## 8. git 사용 규칙

- 커밋 메시지는 한국어로, 변경 내용을 한 줄로 요약한다
- 예: git commit -m "몬스터 HP x1.3 적용, 연쇄고리 너프"
- 작업 전 백업 커밋과 작업 후 완료 커밋을 반드시 분리한다
- 다른 컴퓨터에서 이어 작업할 때는 반드시 git pull부터 한다
- 큰 기능 추가 전에는 브랜치를 만들어서 작업하고, 완료 후 main에 합친다

## 요약: 핵심 3줄

1. **백업 먼저, 수정은 그 다음** -- git commit 없이 코드 수정을 시작하지 않는다
2. **작게 나눠서, 매번 확인** -- 5건 이내 수정, 1건마다 lint 검증, 세션마다 테스트
3. **논의와 구현을 섞지 않는다** -- 설계는 채팅에서, 코딩은 터미널에서
