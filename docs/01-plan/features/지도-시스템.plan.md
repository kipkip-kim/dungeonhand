# Plan: 지도 시스템 (Slay the Spire 스타일)

## Executive Summary

| 관점 | 내용 |
|------|------|
| **문제** | 층당 전투 2회로 단조로움, 경로 선택 없이 일직선 진행 |
| **해결** | 층별 노드 맵 도입 — 전투/이벤트/상점/캠프 노드를 선택하며 진행 |
| **기능/UX 효과** | 전략적 경로 선택, 층당 전투 3~4회로 확장, 리플레이 가치 증가 |
| **핵심 가치** | 매 런마다 다른 경로 → 전략적 깊이 + 리플레이성 |

---

## 1. 현재 구조 분석

### 현재 층별 흐름 (고정)
```
전투1(bn=1) → 전투2(bn=2) → 캠프파이어(bn=3) → 미니보스(bn=4) → 보스(bn=5) → 상점
```
- `battleNum` 1~5로 진행 단계 추적
- `BATTLE_TO_SLOT` = {1:0, 2:1, 4:2, 5:3} — battleNum→몬스터 인덱스 매핑
- `MONSTERS_PER_FLOOR` = 4 (일반2 + 미니보스1 + 보스1)
- `advanceBattle()`: battleNum++ 후 분기 (campfire/battle)

### 변경이 필요한 핵심 구조
| 현재 | 변경 후 |
|------|---------|
| `battleNum` 1~5 고정 | 맵 노드 인덱스로 대체 |
| `BATTLE_TO_SLOT` 고정 매핑 | 노드별 몬스터/이벤트 타입 지정 |
| `advanceBattle()` → battleNum++ | → 맵 화면 표시 → 노드 선택 |
| 캠프파이어 위치 고정 (bn=3) | 맵 노드로 배치 |

## 2. 맵 설계

### 2.1 노드 타입

| 타입 | 아이콘 | 설명 |
|------|--------|------|
| `battle` | ⚔️ | 일반 전투 |
| `elite` | 💪 | 미니보스 전투 (보상 강화) |
| `campfire` | 🔥 | 캠프파이어 (회복+이벤트) |
| `shop` | 🏪 | 상점 |
| `event` | ❓ | 랜덤 이벤트 (신규) |
| `boss` | 👑 | 보스 전투 (층 마지막, 고정) |

### 2.2 층별 맵 구조

```
[시작] → [1열] → [2열] → [3열] → [4열] → [보스]
           ↗ ↘     ↗ ↘     ↗ ↘     ↗ ↘
```

- **열(row)**: 5~6개 (시작 제외, 보스 포함)
- **노드/열**: 2~3개 (선택지)
- **연결**: 인접 열의 노드끼리만 연결 (상/중/하 경로)
- **보스**: 마지막 열에 1개, 모든 경로가 합류

### 2.3 층별 노드 분배 (예시: 6열)

| 열 | 노드 수 | 가능 타입 |
|----|---------|-----------|
| 1 | 3 | battle, battle, event |
| 2 | 3 | battle, campfire, battle |
| 3 | 2 | elite, shop |
| 4 | 3 | battle, event, campfire |
| 5 | 2 | battle, battle |
| 6 | 1 | boss (고정) |

**결과**: 경로당 전투 3~5회 (현재 2회 → 2배 이상)

### 2.4 맵 생성 알고리즘

```
1. 열 개수 결정: 5 (1층) ~ 7 (5층)
2. 각 열에 2~3 노드 생성
3. 노드 타입 배정:
   - 1열: 항상 battle
   - 중간열: battle 60%, campfire 15%, event 15%, shop 10%
   - elite는 열3~4에 1개 보장
   - 마지막 열: boss 1개
4. 연결 생성: 각 노드 → 다음 열의 1~2개 노드 연결
   - 최소 1개 연결 보장 (데드엔드 방지)
   - 교차 연결 허용 (위→아래, 아래→위)
```

## 3. 데이터 구조

### 3.1 맵 데이터 (useState로 관리)

```javascript
// floorMap state
{
  floor: 1,
  rows: [
    // row 0 (start)
    [{ id: "start", type: "start", connections: [0, 1, 2] }],
    // row 1

      { id: "1-0", type: "battle", monIdx: 0, connections: [0, 1], visited: false },
      { id: "1-1", type: "battle", monIdx: 1, connections: [1], visited: false },
      { id: "1-2", type: "event", eventId: "fairy", connections: [1, 2], visited: false },

    // row 2
    [
      { id: "2-0", type: "campfire", connections: [0], visited: false },
      { id: "2-1", type: "battle", monIdx: 2, connections: [0, 1], visited: false },
      { id: "2-2", type: "shop", connections: [1], visited: false },
    ],
    // ...

  currentNode: 0,
```

### 3.2 몬스터 풀 확장

현재 `MONSTERS_PER_FLOOR = 4` (일반2 + 미니보스1 + 보스1)
→ 일반 몬스터를 4~5종으로 확장 필요 (맵에서 3~5회 전투)

**방안 A**: 기존 몬스터 재사용 (같은 몬스터 여러 번)
**방안 B**: 층당 몬스터 5~6종으로 확장 (일반 3~4 + 미니 1 + 보스 1)

→ **방안 A 권장** (방안 B는 20종 → 30종 에셋 필요)

## 4. UI 설계

### 4.1 맵 화면 (새 screen: "map")

```
┌──────────────────────────────┐
│  🗺️ 1층 — 고블린 소굴       │
│                              │
│  [시작] ─── ⚔️ ─── 🔥 ──┐  │
│           ╲              │  │
│            ⚔️ ─── ⚔️ ──── 👑│
│           ╱              │  │
│         ❓ ─── 🏪 ──┘     │
│                              │
│  현재 위치: ●               │
│  방문한 노드: ◉             │
│  선택 가능: ○               │
│  잠김: ◌                    │
└──────────────────────────────┘
```

- 노드: 원형 아이콘, 터치 가능
- 연결선: 실선 (이동 가능) / 점선 (잠김)
- 현재 노드: 하이라이트 + 펄스 애니메이션
- 방문 노드: 어둡게 + 체크마크

### 4.2 흐름 변경

```
현재: 전투 → 보상 → advanceBattle() → 다음 전투/캠프/상점
변경: 전투 → 보상 → 맵 화면 → 노드 선택 → 해당 노드 실행
```

## 6. 대안 검토

### 대안 A: 풀 맵 시스템 (슬더스 스타일)
- 장점: 최대 전략성, 리플레이 가치
- 단점: 구현 복잡도 높음 (2~3세션), battleNum 전면 교체

### 대안 B: 단순 분기점 (전투 후 2갈래)
- 장점: 최소 변경 — advanceBattle에서 2개 선택지 표시
- 단점: 전략 깊이 제한, 슬더스 느낌 부족

### 대안 C: 전투 횟수만 증가 (맵 없이)
- 장점: 가장 단순 — MONSTERS_PER_FLOOR 확장 + battleNum 범위 확장
- 단점: 경로 선택 없음, 요청 의도와 다름

→ **권장: 대안 A** (사용자 요청 의도에 가장 부합)
